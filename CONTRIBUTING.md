# Contributing to pleme-io/actions

Welcome. This repo ships 192+ typed GitHub Actions on **free public
GitHub-hosted CI** — every contribution auto-publishes within minutes
of merge.

## TL;DR — add a new action in 5 minutes

```bash
# 1. Fork + clone
git clone git@github.com:<you>/actions
cd actions

# 2. Scaffold from the simplest sibling (slack-notify is a great template)
cp -r slack-notify my-new-action

# 3. Edit my-new-action/action.yml — inputs, outputs, description, branding
# 4. Edit my-new-action/run.tlisp — your logic (use _tlisp-stdlib helpers)

# 5. Lint locally
diff <(cat _tlisp-stdlib/stdlib.tlisp my-new-action/run.tlisp | python3 -c '
import sys
s = sys.stdin.read()
# ... paren balance check ...
')

# 6. Push + open PR
git add my-new-action/
git commit -m "feat: my-new-action — <one-line description>"
git push origin feature/my-new-action
gh pr create
```

On merge: `auto-bump` fires, tags `v0.13.{next}`, `release.yml`
publishes container images for the 4 Docker-backed actions +
fast-forwards the `v1` ref. **Your action is live + reachable at
`pleme-io/actions/my-new-action@v1` immediately.**

## The canonical pattern

Every action follows the [`pleme-io-pattern-core`](https://github.com/pleme-io/blackmatter-pleme/blob/main/skills/pleme-io-pattern-core/SKILL.md)
shape:

- **action.yml**: 5 canonical sections (name / description / branding /
  inputs / outputs / runs)
- **run.tlisp**: 5-step shape (resolve / validate / detect / execute / branch)
- **README.md**: auto-generated by `pleme-doc-gen` — DO NOT hand-edit

## The hard rules

1. **NO SHELL** in `run.tlisp` body — logic is typed Lisp wrapping
   CLI tools via `exec-capture`. Install steps in `action.yml` may
   use first-party GH actions + the 9-line stdlib loader; that's it.
2. **No duplicated helpers** — if you copy/paste a function from
   another action, extract it to `_tlisp-stdlib/stdlib.tlisp` and
   reference from both.
3. **Inputs are kebab-case** + match shikumi config schema in
   `.pleme-io-release.toml`.
4. **Outputs are documented** — every output earns its name.
5. **Composite balance = 0** — `tlisp-lint` runs on every PR.

CI gates that catch violations:

- [`tlisp-lint`](./tlisp-lint/) — paren/string/comment balance
- [`action-shell-lint`](./action-shell-lint/) — NO-SHELL enforcement
- `regenerate-catalog` — catalog stays in sync
- `enforce-no-shell` — directive lock-in

## What lands automatically

After your PR merges:

- ✅ `v0.13.{next}` tag created in <10s
- ✅ Catalog (`patterns-full.nix`) auto-regenerated
- ✅ Per-action README auto-generated
- ✅ Root index auto-updated
- ✅ Image (if Docker-backed) auto-built + pushed to ghcr
- ✅ `v1` ref auto-fast-forwarded

No version bumps, no release commands, no manual publish steps.

## When to ask vs ship

**Just ship**: new primitive following the pattern, stub body, real
implementation in a follow-up PR.

**Ask first** (open a discussion):

- New _tlisp-stdlib helper — affects every action that consumes it
- New category in the catalog map — affects classification
- Changes to the canonical pattern itself — affects every future action
- New substrate reusable workflow — affects every adopter
